# Phase 05.2: Add Microphone Toggle Button to prevent acoustic leakage - Research

**Researched:** 2026-03-09
**Domain:** Electron app UI + WebRTC MediaStream audio control
**Confidence:** HIGH

## Summary

This phase adds a microphone toggle button to the Listen view UI to prevent acoustic leakage when users only want transcription from system audio. The implementation requires:

1. **UI Component**: Add toggle button in ListenView.js bar-controls (after Save button)
2. **State Management**: Track microphone enabled/disabled state in ListenView
3. **Audio Control**: Enable/disable microphone MediaStream tracks in listenCapture.js

The codebase already has all necessary infrastructure: LitElement for UI, MediaStream API for microphone control, and a clear communication path between UI and audio capture modules.

**Primary recommendation:** Add `micEnabled` property to ListenView, create toggle button with microphone icon, and expose mic toggle functions in listenCapture.js that can be called from the UI component.

**Add toggle method:**
```javascript
toggleMic() {
    this.micEnabled = !this.micEnabled;
    if (window.listenCapture) {
        if (this.micEnabled) {
            window.listenCapture.enableMic();
        } else {
            window.listenCapture.disableMic();
        }
    }
    this.requestUpdate();
}
```

### 2. listenCapture.js Changes

**Add toggle functions:**
```javascript
// Global state to track if mic was previously enabled
let micWasEnabled = true;

function disableMic() {
    if (micMediaStream) {
        micMediaStream.getTracks().forEach(track => track.stop());
        micMediaStream = null;
    }
    if (audioProcessor) {
        audioProcessor.disconnect();
        audioProcessor = null;
    }
    if (audioContext) {
        audioContext.close();
        audioContext = null;
    }
    console.log('Microphone disabled');
}

async function enableMic() {
    if (micMediaStream) {
        console.log('Microphone already enabled');
        return;
    }

    try {
        micMediaStream = await navigator.mediaDevices.getUserMedia({
            audio: {
                sampleRate: SAMPLE_RATE,
                channelCount: 1,
                echoCancellation: true,
                noiseSuppression: true,
                autoGainControl: true,
            },
            video: false,
        });

        if (isMacOS) {
            const { context, processor } = await setupMicProcessing(micMediaStream);
            audioContext = context;
            audioProcessor = processor;
        } else if (isLinux) {
            setupLinuxMicProcessing(micMediaStream);
        } else {
            const { context, processor } = await setupMicProcessing(micMediaStream);
            audioContext = context;
            audioProcessor = processor;
        }
        console.log('Microphone enabled');
    } catch (err) {
        console.error('Failed to enable microphone:', err);
    }
}

// Expose to window
window.listenCapture = {
    ...module.exports,
    disableMic,
    enableMic,
};
```

**Note:** Platform-specific handling (macOS vs Linux vs Windows) is needed because:
- macOS uses `setupMicProcessing()` with AEC
- Linux uses `setupLinuxMicProcessing()` without AEC
- Windows uses `setupMicProcessing()` with AEC

## Common Pitfalls

### Pitfall 1: Microphone not re-enabling properly
**What goes wrong:** After disabling mic, clicking toggle doesn't restart microphone capture
**Why it happens:** The microphone MediaStream is completely stopped and nullified, but re-enablement code may fail silently or not handle platform differences
**How to avoid:** Ensure enableMic() properly re-initializes the audio context and processor for each platform
**Warning signs:** Console shows "Failed to enable microphone" errors

### Pitfall 2: State not updating in UI
**What goes wrong:** Button visual state doesn't reflect actual microphone state
**Why it happens:** Not calling `this.requestUpdate()` after toggling
**How to avoid:** Always call requestUpdate() after modifying reactive properties

### Pitfall 3: AudioContext not closed on disable
**What goes wrong:** Memory leak or audio conflicts on re-enable
**Why it happens:** Only stopping tracks but not closing AudioContext
**How to avoid:** Properly clean up audioContext, audioProcessor, and micMediaStream

### Pitfall 4: Toggle during active session
**What goes wrong:** Toggle doesn't work if session just started
**Why it happens:** Race condition between session start and mic initialization
**How to avoid:** Check for null micMediaStream in both enable and disable functions

## Code Examples

### MediaStream Track Control (Standard Web API)
```javascript
// Stop all tracks in a MediaStream
function stopMicrophone(micMediaStream) {
    if (micMediaStream && micMediaStream.getTracks) {
        micMediaStream.getTracks().forEach(track => {
            track.stop();
            console.log(`Stopped track: ${track.kind}`);
        });
    }
}

// Check if microphone is currently active
function isMicActive(micMediaStream) {
    return micMediaStream &&
           micMediaStream.getTracks().length > 0 &&
           micMediaStream.getTracks().some(track => track.kind === 'audio' && track.enabled);
}
```

Source: [MDN MediaStream API](https://developer.mozilla.org/en-US/docs/Web/API/MediaStream)
